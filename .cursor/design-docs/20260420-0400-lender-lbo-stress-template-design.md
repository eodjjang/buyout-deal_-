## Problem Statement

인수금융(Buyout) 딜 심사역은 매주~격주 진행 중인 딜에서 **매수자(또는 매수자 측 자문사)가 작성한 LBO 모델**을 받아, 대주단 관점의 보수적 재검증을 수행한 뒤 심사보고서(Word)의 **Valuation 적정성** 및 **Exit 가능성(이자지급·만기상환)** 파트를 작성해야 한다. 한 건당 해당 파트 작성에 **2~10 영업일**이 소요되며, 그 중 가장 큰 병목은 **"매수자 모델에 Base/Up/Down 시나리오 switch를 심어 스트레스 테스트된 별도 LBO 모델을 재구축하는 작업"**이다.

매수자 모델은 딜마다 시트 이름·셀 위치·계산식이 제각각이라 매번 파일 복제→수작업 개조가 반복되고, 그 결과물의 품질·재현성·심사 표준성이 낮다. 본 설계는 이 병목을 해소할 **대주단 자체 범용 LBO 표준 템플릿(MVP)**의 엑셀 구조·로직·파싱 규약을 정의한다.

## Demand Evidence

- 본인이 매주~격주 진행 중인 딜에서 직접 2~10일을 태운다 (Weekly pain)
- 여러 후보 병목 중 "스트레스 시나리오 반영된 별도 LBO 모델을 다시 만드는 작업"이 가장 시간을 잡아먹는다고 특정됨 (추상적 "모두 오래 걸린다" 답변에서 최근 딜 회고 push를 통해 확정)
- 산출물은 본인 1명 사용 → 팀장 보고 라인까지. 외부 공유 없음

## Status Quo

1. 매수자 LBO 모델(.xlsx) 수령 → 원본 보존용 사본 생성
2. 시트를 훑어 핵심 드라이버(매출, EBITDA, Capex, NWC) 위치 수작업 식별
3. 별도 탭 또는 별도 파일에 스트레스 가정을 손으로 타이핑
4. 주요 수식을 수작업으로 재지정(=기존 셀 × 스트레스 팩터)
5. Debt schedule, Cash flow waterfall, DCF를 다시 각자 구축
6. 계산 결과를 Word 심사보고서의 표와 코멘트로 옮김
7. 상사 피드백 → 2~5 반복

각 딜마다 파일 구조가 달라서 **전 딜의 템플릿이 다음 딜에 거의 재사용되지 않는다**. 심사 표준성·재현성·감사추적성이 낮다.

## Target User & Narrowest Wedge

- **Primary user**: 대주단 심사역 본인 (solo). 최종 소비자는 팀장 검토 라인.
- **Narrowest wedge**: 매수자 모델에서 **최소 4개 드라이버만 붙여넣으면** Base/Up/Down 시나리오가 자동 계산되고 Opco/Holdco 부채 스케줄, LTV, DSCR/ICR이 산출되는 **단일 .xlsx 템플릿**.
- **명시적 제외(v2+)**: Peer 멀티플 자동 조회, DCF 상세 민감도, Word 자동 생성, 크로스-딜 데이터베이스. 단, 이들이 나중에 붙을 **consume 포인트(Dashboard 시트 + 표준 Named Range)**는 MVP 단계에서 선점해 둔다.

## Constraints

- **Tech**: Pure Excel .xlsx (매크로·VBA 지양. Office 365 내장 함수만 사용 → 사내 PC 호환성·보안)
- **Fill 컨벤션**: 다크블루 섹션 헤더(#1F4E79) + 라이트블루 칼럼 헤더(#D9E1F2) + 그레이 입력(#F2F2F2) + 미디엄블루 Key output(#BDD7EE)
- **Font 컨벤션**: 입력 파란색(0000FF), 계산 검정(000000), 같은 탭 링크 보라(800080), 타탭 링크 녹색(008000), **CIQ Plug-in 수식 청록(008B8B) — v0.4 신설**
- **Sign convention**: 모든 지출·차감 항목은 **양수로 표기하고 수식에서 차감(-)** (GAAP-style) — 전 시트 일관 적용
- **영구성장률**: 1.0% 고정, 민감도 없음 (사용자 지정)
- **단위**: KRW 백만원 단일 단위. USD 딜은 환산 후 입력. 단위 혼용 금지.
- **회계연도**: 대상기업 FY 그대로 사용. CY 변환은 주석으로만.

## Phase 0 — Preconditions (템플릿 착수 이전 반드시 해결) [v0.4 신설, v0.5에서 3건 모두 해결]

아래 3건 중 하나라도 미해결 상태에서 `1_Input_BaseCase` 이후 시트를 구현하기 시작하면 **후행 재작업이 불가피**합니다. v0.5 시점 **3건 모두 ☑ 완료** → 엑셀 템플릿 셀 작성 단계 진입 가능.

### Precondition 1 — Lender-Adjusted EBITDA 정의 ☑ 해결 (v0.5)

- **왜 blocker인가**: 본 템플릿의 `1_Input!EBITDA`가 입력 기준으로 삼는 값의 정의가 불명이면, 스트레스·DSCR·ICR·LTV 모든 하류 숫자의 해석이 본부 기준과 mismatch될 수 있음.
- **사용자 답변 (v0.5)**: **"모든 add-back 불인정"** — 일회성 비용, SBC, 오너 보상, 사외이사 보수, 일시적 CAPEX 효과, 인수 관련 자문비 등 **일체 인정하지 않음**.
- **설계 반영**:
  - `1_Input!EBITDA`는 **Reported EBITDA 단일**. Adjusted EBITDA 행 제거.
  - `1_Input`에 있던 add-back 항목별 입력 행 제거.
  - Guardrail #2를 "Reported EBITDA만 추출. Adjusted는 비고 주석"으로 간소화.
  - 매수자가 Adjusted 기준으로 작성한 모델이면 **add-back을 역가산하여 Reported로 환원 후 입력** (Assignment 1 드라이런 체크포인트).
- **상태**: ☑ 해결 (2026-04-20)

### Precondition 2 — 사내 Word 심사보고서 표준 양식 확보 ☑ 해결 (v0.5)

- **왜 blocker인가**: `8_Dashboard`의 표 구성·셀 순서·Named Range 이름이 Word 양식의 표와 매칭되어야 v2 Word 자동화가 compatible.
- **사용자 제출 (v0.5)**: `d:\영빈\영빈2\심사보고서\[제2023-23차 투자심사운영위원회]심사보고서_HMM 인수금융 입찰용 LOC 발급의 건_F.docx` (1,829 paragraph / 2.3MB, 에이전트가 자동 파싱).
- **추출된 Valuation 파트 표 구조**:
  1. **Summary 판정 행**: 4-scale (긍정/다소긍정/다소부정/부정) 체크.
  2. **담보 지분가치 비교표** (5열 × 10행): 구분 | 대상회사 주가 | 상대가치(PBR/EV·EBITDA) | DCF | 비고. 행 = 순자산·Multiple·지분가치(100%)·지분율·담보지분가치·인수금융·본건 LTV·(참고)완전희석 LTV.
  3. **Peer Multiple 산정내역**: Peer명·순자산·시총·Multiple·Outlier 여부.
  4. **DCF FCFF 테이블** (11열, 10Y): 매출 → EBITDA → Tax → CAPEX → NWC → FCFF → PV of FCFF → Total PV → PV of TV → EV → Equity.
  5. **스폰서 vs 심사팀 EBITDA 비교** (연도별).
- **추출된 Exit/원리금 상환 파트 표 구조**:
  1. **자금수지표** (5열, 4~5Y): 기초현금 → 영업CF → 투자CF(CAPEX) → 배당수익 → 재무CF → 본건 인수금융 이자/원금 → 기말현금.
  2. **만기 Exit Value 표**: Exit Multiple × 순자산 × 지분율 = Implied Exit Value.
- **설계 반영** (자세한 사항은 §7, §8 참조):
  - `7_Returns_LTV`: 평가방식 고정 3카테고리 해제 → "평가방식 1/2/3" 추상화 (Q1-C).
  - `8_Dashboard`: `DASH_CFTable_*` Named Range 그룹 신설 (Q3). 자금수지표 8행 × 최대 5개년 형태.
  - `6_DCF_Valuation`: 스폰서 input 열 + 심사팀 stressed 열 **이중 병기 구조** 추가 (Word의 "스폰서 vs 심사팀 EBITDA 비교" 대응).

## Premises (검증 결과)

- **P1 — 매수자 모델 구조 다양성**: ✅ 동의. 전체 파싱은 포기. 표준 템플릿 + 4개 드라이버 파싱 + 오버레이 스트레스 채택.
- **P2 — Peer 데이터 소스**: ✅ v0.3/v0.4 확정. 사용자 계정에 **CapIQ Pro 웹 + Office Plug-in**이 설치되어 있음. `=CIQ(...)` / `=CAPIQ(...)` 수식을 템플릿 셀에 직접 박아 실시간 자동 조회를 primary로 사용. 딜당 사람 개입은 **Ticker 리스트 입력 1회 + `Data → Refresh All` 1회 = 총 2-action**. 단, Precondition 3 검증 결과에 따라 설계 조정 가능.
  - **Trading Peers**: Plug-in 수식으로 Ticker → Multiple 자동 조회 (coverage 우수)
  - **Transaction Comps**: Plug-in의 `=CIQTRANSACTION(...)` 계열로 1차 조회 가능하나, 한국 비상장 M&A 거래의 Deal Value 미공개·미등재 문제는 여전. **Plug-in 1차 + Kisvalue/한경 Compass/투자조선 수기 보완**의 이중 소스 구조 유지. `9c_Manual_Supplement` 시트 계속 사용.
  - **Fallback 호환**: Plug-in 장애·라이선스 만료 등 예외 상황에서 Export+Paste 워크플로우 병행 지원. v0.5에서 Alt-A (동일 셀 택1) 확정 — §9 intro 참조.
- **P3 — 기초 자료 스키마**: ✅ 암묵 동의. MVP에서는 `8_Dashboard` 시트를 **고정 셀 주소 + Named Range**로 강제하여 추후 JSON export의 원천으로 삼는다. Named Range 명명 규칙은 §8 참조.
- **P4 — Solo 사용자 반자동 템플릿이 ROI 최선**: ✅ 동의. Word 자동 생성은 v2+.
- **P5 — 보수성은 정량 규칙으로 고정**: ✅ 동의. `2_Stress_Panel`에 산업 디폴트 프리셋을 내장해 사용자가 override만 하도록 강제.

## Approaches Considered

### Approach A — Parser-driven Full Template (대안, 비추천)

- 매수자 모델 전체를 LLM/스크립트로 파싱해 표준 템플릿에 자동 매핑.
- Summary: 사용자가 매수자 파일을 업로드하면 드라이버가 자동 추출되어 템플릿에 꽂힌다.
- Effort: **XL** (매수자 모델 variance가 너무 커서 테스트 케이스 20+ 필요)
- Risk: **High** (오탐·누락 시 심사 신뢰도 치명)
- Pros: 완전 자동화 시 시간 절감 최대
- Cons: Pre-product 단계에서 검증 불가. 실패하면 뒷단 전부 쓸모없음.
- Reuses: 없음 (from scratch)

### Approach B — Paste-driven MVP with Overlay Stress + CapIQ Plug-in ⭐ Recommended

- **운영 드라이버**: 매수자 모델에서 **4개 드라이버(Revenue, EBITDA(or Margin), Capex, NWC)**만 사람이 Paste Special (Values)로 붙여 넣으면 스트레스 엔진이 자동 연산.
- **Peer 데이터**: **CapIQ Office Plug-in 직접 수식이 primary**. `9a`/`9b`에 `=CIQ(...)` 수식을 미리 박아두고 **Ticker 입력 + `Data → Refresh All`** 만으로 Trading/Transaction 멀티플이 실시간 갱신. `9_Peer_Summary`의 Include ✓ 토글로 최종 확정. Plug-in 비가용 시 Export+Paste 경로로 fallback.
- Summary: **"파싱"은 사람이 눈으로(4대 드라이버) + "Peer"는 Plug-in이 실시간**.
- Effort: **M+** (2~3주. CapIQ 통합 시트 설계에 1주 추가)
- Risk: **Low-Med**
- Pros:
  - MVP가 **혼자서도 끝까지 완성 가능** (VBA·외부 API 불필요)
  - 매수자 모델 variance에 내성
  - CapIQ 통합으로 **Peer/Transaction 작업 시간 ~80% 단축** 기대 (기존 수기 2~4h → 3~5min Plug-in / 15~25min Paste)
  - Dashboard 고정 주소 덕에 v2(Word 자동 생성, 다른 데이터 소스 추가) 로드맵과 compatible
- Cons:
  - 4대 드라이버 추출은 여전히 수작업 (딜당 15~30분 예상)
  - 운영 모델의 **상세 라인 아이템 스트레스**는 불가 (SG&A만 따로 흔드는 식의 드릴다운 없음). 대주단 심사 수준에서는 4대 드라이버 오버레이로 충분.
  - CapIQ Export 컬럼 순서 변경 시 헤더 재매핑 필요 (월 1회 정기 점검)
  - **한국 비상장 M&A Transaction Comps는 CapIQ 단일 소스로 부족** → `9c_Manual_Supplement` 수기 보완 필수
- Reuses: 기존 `lbo-model` 스킬 템플릿(`examples/LBO_Model.xlsx`) 구조를 레퍼런스로 활용. Saved Screen은 1회 세팅 후 전 딜 재사용.

### Approach C — External Scenario Wrapper (대안, 제한적)

- 매수자 원본 모델을 건드리지 않고, 별도 Wrapper 파일에서 `INDIRECT()` / 외부 링크로 주요 셀을 참조하여 스트레스 팩터를 곱함.
- Effort: **S**, Risk: **Med**
- Pros: 빠른 착수, 원본 무결성 보존.
- Cons: 재사용성 낮음, Dashboard 표준화 불가, DCF·Waterfall 모듈 얹기 어려움 (결국 B로 회귀).

### **RECOMMENDATION: Approach B**

Pre-product 단계에서 **혼자 끝까지 완성 가능한 최소 단위**인 동시에, v2+ (Peer DB, DCF 상세, Word 자동화)와 가장 compatible한 단 하나의 선택지.

### 1. `1_Input_BaseCase` — 매수자 모델로부터 붙여 넣는 시트

**목적**: 사용자가 매수자 모델에서 눈으로 식별한 4대 드라이버 + 인수 조건을 **값만(Paste Special Values)** 붙여 넣는 단일 진입점.

#### 1.1 Section A — 인수 조건 (Transaction Terms)

| Row | 항목 | 단위 | 비고 |
|-----|------|------|------|
| 인수금액 (Purchase EV) | KRW MM | 블루(입력) |
| Less: Net Debt Assumed | KRW MM | 블루 |
| = 지분 인수가액 (Equity Purchase Price) | KRW MM | 계산 |
| + Transaction Fee (M&A 자문·실사·세무) | KRW MM | 블루 |
| = Uses of Funds 합계 | KRW MM | 계산 |
| Sources: Opco Senior TL | KRW MM | 블루 |
| Sources: Opco 2nd Lien | KRW MM | 블루 |
| Sources: Holdco Sub Loan | KRW MM | 블루 |
| Sources: Sponsor Equity | KRW MM | 계산 (plug) |
| Target Net Debt / LTM EBITDA (본부 승인치) | x | 블루 (예: 6.0x) |
| Closing Date | Date | 블루 |
| Exit Date (Assumed) | Date | 블루, MVP는 Closing + 5Y 디폴트 |

> **Check rows (v0.4 이중 구조)**:
> 1. `Sources 합계 − Uses 합계 = 0` — Sponsor Equity가 plug인 이상 항등식이므로 **표시용 sanity row**(실 검증 가치 낮음, 빨강 조건부 서식 유지하되 의사결정 신호로 쓰지 않음).
> 2. **`(Opco Senior + Opco 2nd Lien + Holdco Sub) / LTM_EBITDA ≤ Target_Leverage`** — 실질 검증. 초과 시 빨강. Sponsor Equity plug가 과소 산정되어 Debt가 Target을 넘는 경우를 잡는다.

#### 1.2 Section B — Base Case 4대 드라이버 (FY 기준)

시점 축: `FY-2 Actual / FY-1 Actual / LTM / FY1 / FY2 / FY3 / FY4 / FY5` (총 8개 컬럼). 컬럼 헤더는 매수자 모델 기준 회계연도 그대로.

| Row | 항목 | 비고 |
|-----|------|------|
| Revenue | 매수자 Base Case 그대로 |
| COGS 또는 Gross Profit | 둘 중 매수자 모델 구조에 맞는 것 하나만 |
| SG&A | 선택적. 없으면 EBITDA에서 역산 |
| **EBITDA (Reported)** | **v0.5: Reported EBITDA 단일**. 매수자 모델이 Adjusted 기준이면 add-back 금액을 역가산해 Reported로 환원 후 입력 (본부 기준 "모든 add-back 불인정", Precondition 1). 매수자 원 Adjusted 값은 주석에만 병기 |
| D&A | 이후 Tax on EBIT 계산용 |
| Capex | Maintenance + Growth 구분되어 있으면 별도 2행 |
| Δ NWC (증가 = 현금유출, +로 표기) | 부호 주의 (아래 에이전트 조언 참조) |
| Effective Tax Rate | Blended rate, % |

> **v0.4 변경**: v0.3에서 추가되었던 `[Optional] P — Blended Price`, `[Optional] Q — Total Volume` 행은 v1.1 backlog로 연기. § Deferred Backlog 참조.

#### 1.3 Section C — Implied 역산 지표 (검증용)

매수자 숫자를 붙이고 나면 **템플릿이 자동으로 역산**해서 사용자가 눈으로 sanity check.

- EBITDA Margin = EBITDA / Revenue
- Capex as % of Revenue
- ΔNWC as % of ΔRevenue (또는 % of Revenue)
- Revenue YoY Growth %
- EBITDA YoY Growth %

이들을 **Base Case 암묵 가정**으로 간주하고, `2_Stress_Panel`의 스트레스는 이 지표들을 기준으로 shock을 가한다.

### 2. `2_Stress_Panel` — 하이레벨 오버레이 스트레스 패널

#### 2.1 Case Switch

- 셀 `B3` (Named range `Case_Switch`): 데이터 유효성 검증 드롭다운 — `Base` / `Upside` / `Downside`
- 이 셀 하나가 바뀌면 `3_Operating_Overlay` 이후 모든 시트 숫자가 바뀌도록 설계 (`CHOOSE`, `XLOOKUP`, 또는 `SWITCH` 함수 기반)

#### 2.2 3열 스트레스 파라미터 테이블

| 파라미터 | Base | Upside | Downside | 단위 | 적용 대상 |
|----------|------|--------|----------|------|-----------|
| Revenue Growth Δ | 0.0%p | +2.0%p | -5.0%p | %p (매 연도 매출성장률에 가산) | `3_Overlay!Revenue` |
| EBITDA Margin Δ | 0.0%p | +0.5%p | -3.0%p | %p | `3_Overlay!EBITDA` |
| Capex % of Revenue Δ | 0.0%p | -0.5%p | +1.5%p | %p | `3_Overlay!Capex` |
| ΔNWC % of Revenue Δ | 0.0%p | 0.0%p | +1.0%p | %p | `3_Overlay!NWC` |
| WACC Uplift | 0 bp | -50 bp | +100 bp | bp | `6_DCF!WACC` |
| Exit Multiple Δ | 0.0x | +1.0x | -1.0x | turn | `7_Returns!ExitMultiple` |
| Permanent Growth | 1.0% | 1.0% | 1.0% | 고정 |
| 산업 디폴트 프리셋 | — | — | — | 드롭다운(소매/제조/SaaS/헬스케어/...) → 위 값 자동 채움 |

> **v0.4 변경**: v0.3의 `Price 변화율(%)` · `Volume 변화율(%)` 파라미터 2행 및 `PQ_Mode` 플래그·조건부 서식(ACTIVE/IGNORED 배지)은 v1.1 backlog로 연기. § Deferred Backlog 참조.

#### 2.3 활성 시나리오 추출

각 파라미터 오른쪽에 `Active` 컬럼 추가:
- 예: `Active_Revenue_Δ = CHOOSE(MATCH(Case_Switch, {"Base","Upside","Downside"},0), Base, Upside, Downside)`
- 또는 `=SWITCH(Case_Switch, "Base", B8, "Upside", C8, "Downside", D8)`

이 `Active_*` 열이 단일 진실 원천(Single source of truth)이 되어 `3_Operating_Overlay`에서 참조된다.

### 3. `3_Operating_Overlay` — 스트레스된 운영 지표 계산

시점 축은 `1_Input_BaseCase` Section B와 동일 (FY-2 ~ FY5).

| Row | 공식 |
|-----|------|
| Base Revenue | `='1_Input_BaseCase'!Revenue` (녹색 링크) |
| Base YoY Growth | `= Base_Rev[t] / Base_Rev[t-1] - 1` |
| Stressed YoY Growth | `= Base_YoY + Active_Revenue_Δ` (FY1 이후만 적용, Actual 기간은 pass-through) |
| **Stressed Revenue** | `= Stressed_Rev[t-1] × (1 + Stressed_YoY[t])` |
| Base EBITDA Margin | `='1_Input_BaseCase'!EBITDA / Revenue` |
| **Stressed EBITDA Margin** | `= Base_Margin + Active_Margin_Δ` |
| **Stressed EBITDA** | `= Stressed_Rev × Stressed_Margin` |
| Base Capex % of Rev | `='1_Input_BaseCase'!Capex / Revenue` |
| **Stressed Capex** | `= Stressed_Rev × (Base_Capex% + Active_Capex_Δ)` |
| Base ΔNWC % of Rev | `='1_Input_BaseCase'!ΔNWC / Revenue` |
| **Stressed ΔNWC** | `= Stressed_Rev × (Base_NWC% + Active_NWC_Δ)` |
| D&A | Base pass-through (스트레스 미적용. 보수적 논거: D&A는 회계항목) |
| EBIT (Stressed) | `= Stressed_EBITDA - D&A` |
| Cash Taxes | `= MAX(0, Stressed_EBIT) × Effective_Tax_Rate` (NOL 이월 없는 보수적 가정) |
| **UFCF (Stressed)** | `= Stressed_EBITDA - Cash_Taxes - Stressed_Capex - Stressed_ΔNWC` |

> **v0.4 변경**: v0.3에서 병렬 설계되었던 *P·Q 경로*(Base P / Base Q / Stressed P / Stressed Q / P·Q path Stressed Revenue) 및 `IF(PQ_Mode, ..., ...)` 분기, ±30% sanity 배너는 모두 v1.1 backlog로 연기. MVP는 Revenue Growth 단일 경로. § Deferred Backlog 참조.

### 4. `4_Debt_Schedule` — Opco·Holdco 부채 스케줄

3개 트랜치 각각 아래 구조 반복:

| Row | Opco Senior TL | Opco 2nd Lien | Holdco Sub |
|-----|----------------|---------------|------------|
| Opening Balance | FY1에 Sources 금액, 이후 전기 Ending | 동일 | 동일 |
| Interest Rate | 입력 (블루) | 입력 | 입력 |
| Interest Expense | = Opening × Rate (순환참조 방지를 위해 **기초잔액** 사용) | 동일 | 동일 |
| Mandatory Amortization | 입력 스케줄 (블루) or % of opening | % of opening | 보통 bullet = 0 until maturity |
| Cash Sweep (Opco only) | Priority 1 | Priority 2 | — |
| Ending Balance | `= MAX(0, Opening − Mand − Sweep)` | 동일 | 동일 |
| Maturity Check | 연도 ≥ maturity 이면 잔액 → bullet repayment로 전액 상환 시도 |

**Cash Sweep 우선순위**: Opco UFCF가 mandatory amort와 이자를 전부 충당하고 남으면 **Sweep %**(입력)만큼 Senior TL부터 prepayment, 그 다음 2nd Lien.

**Holdco는 이자만 PIK or Cash-Pay**. 사용자가 PIK/Cash 선택 가능한 드롭다운. PIK면 Interest가 Ending Balance에 가산. (**Holdco 원금 Sweep은 v1.1 backlog** — § Deferred Backlog 참조)

### 5. `5_CF_Waterfall` — 현금흐름 폭포

| Row | 공식 |
|-----|------|
| Opco UFCF | `='3_Overlay!UFCF` |
| Less: Opco Interest (Senior + 2nd Lien) | `='4_Debt!Opco_Interest_합계` |
| Less: Opco Mandatory Amort | `='4_Debt!Opco_Mand_합계` |
| = Opco CFADS (Cash Flow Available for Dist.) | |
| Less: Minimum Cash Retention | 입력 (블루, 보수값 ex. 1개월 매출) |
| Less: Legal Reserve 적립 | 입력 (이익준비금 10%) |
| = **Distributable to Holdco** | `MAX(0, CFADS − Min Cash − Legal Reserve)` |
| × 배당성향(Payout Ratio) | 입력 (보수적 = 100%, 단 Distributable Retained Earnings 한도 내) |
| = **Dividend Paid to Holdco** | |
| Holdco: Dividend Received | |
| Less: Holdco Interest (if Cash-Pay) | `='4_Debt!Holdco_Interest` |
| = **Holdco Net Cash Flow** | |
| Holdco Cash Balance (Ending) | |
| **Interest Coverage Check (Holdco)** | `Dividend Received / Holdco Interest ≥ 1.0x` 조건부 서식 빨강 |

**핵심 KPI** (`8_Dashboard`로 export):
- Opco DSCR = CFADS / (Opco Interest + Mand Amort)
- Opco ICR = EBITDA / Opco Interest
- Holdco ICR = Dividend Received / Holdco Interest
- Net Leverage = (Opco Debt + Holdco Debt) / LTM EBITDA

### 6. `6_DCF_Valuation` — 스트레스 하의 DCF

사용자 요구사항의 표 구조(총 현금유입/유출 → FCFF → 할인 → PV → EV → 담보가치 → 자기자본가치) 그대로 반영.

| Row | FY1 | FY2 | FY3 | FY4 | FY5 | TV |
|-----|-----|-----|-----|-----|-----|-----|
| Stressed EBITDA | 링크 | … | | | | |
| (-) Cash Taxes on EBIT | 링크 | | | | | |
| (-) Capex | 링크 | | | | | |
| (-) ΔNWC | 링크 | | | | | |
| **FCFF** | | | | | | |
| WACC | = Base WACC + Active_WACC_Δ | | | | | |
| Discount Period | **0.5 / 1.5 / 2.5 / 3.5 / 4.5** (mid-year) | | | | | **5.0** (year-end) |
| Discount Factor | `= 1/(1+WACC)^Period` | | | | | |
| PV of FCFF | | | | | | |
| Terminal Value (Gordon) | `= FCFF_FY5 × (1+1%) / (WACC − 1%)` (FY5 말 기준값) | | | | | |
| PV of TV | `= TV / (1+WACC)^5.0` | | | | | |
| **EV (PV 합계)** | | | | | | |
| (+) 비영업자산 (사용자 입력) | | | | | | |
| (-) Net Debt (Closing) | | | | | | |
| **= 담보기준 Equity Value (for LTV)** | | | | | | |

> **v0.4 TV 할인 기간 교정**: v0.3에서 TV 할인 기간을 4.5로 표기했으나, Gordon growth model의 TV는 FY5 말 기준 "FY6부터 영구"의 현재가치이므로 할인 시점은 **5.0 (FY5 year-end)**가 맞다. FCFF의 mid-year(0.5~4.5)와 TV의 year-end(5.0)는 의도된 비대칭이며 교과서적 convention.

영구성장률 1.0% 고정·민감도 미적용은 사용자 명시 요구사항.

### 에이전트를 위한 조언 (Parsing Guardrails)

매수자 LBO 모델에서 4대 드라이버를 추출할 때 LLM 에이전트가 반드시 점검할 사항:

1. **NWC의 부호(Sign) 확인** — 가장 흔한 오류
   - 매수자가 "Change in NWC"를 **증가=양수(자산 증가=현금유출)**로 쓰는지, **현금흐름표 관점 음수**로 쓰는지 확인
   - 본 템플릿은 **증가=양수(현금유출)** 컨벤션. 반대 부호면 `−1`을 곱해 입력
   - 검증: Working Capital roll-forward가 있으면 ΔNWC = NWC[t] − NWC[t-1] 로 직접 재산출해 대조

2. **Reported EBITDA 단일 추출 (v0.5 간소화)**
   - 본부 기준 **모든 add-back 불인정** (Precondition 1 해결 결과). `1_Input!EBITDA`는 Reported EBITDA만 입력.
   - 매수자 모델이 Adjusted EBITDA 기준이면:
     - add-back 항목 리스트 (일회성 비용·SBC·오너 보상·자문비 등)를 눈으로 식별
     - 각 add-back 금액의 **합계를 역가산**해 Reported로 환원
     - 환원식: `Reported EBITDA = Adjusted EBITDA − Σ(모든 add-back 금액)`
   - 매수자 원 Adjusted 값과 add-back 리스트는 **`1_Input!Note` 셀에 주석으로 병기**(감사추적성).
   - 매수자 모델이 이미 Reported 기준이면 그대로 입력 + "Reported" 명시.

3. **Management Case vs Bank Case 구분**
   - 매수자 모델에 여러 Case가 있으면 (Mgmt / Bank / Stretch), **Bank Case가 이미 있는 경우에도 대주단 독립 스트레스의 기준(Base)은 Management Case**로 채택하는 것이 보수적 (Bank Case는 이미 한 번 깎인 숫자이므로 이중 스트레스가 되어 지나치게 보수적)
   - 단, 본부 정책에 따라 Bank Case를 Base로 쓰는 경우도 있음 → 사용자 선택 후 `1_Input!Note`에 기록

4. **Transaction Fee 처리**
   - 매수자 모델에서 Fee가 **P&L로 비용 처리**되는지(영업외비용) **차입 규모 산정에 포함**되는지 확인
   - 본 템플릿은 Fee를 **Uses of Funds에만 포함**하고 P&L 스트레스에는 미반영 (Day-1 비용)

5. **Fiscal Year vs Calendar Year**
   - 한국 기업은 대개 CY = FY지만, 12월 결산이 아닌 기업도 있음
   - 에이전트는 매수자 모델의 **첫 컬럼 헤더 문자열(예: "FY22", "FY24E", "2024E")을 그대로 추출**해 `1_Input` 컬럼 헤더에 복제. 임의 해석 금지

6. **Interest Expense Sign Convention**
   - 매수자 모델에서 Interest Expense가 **음수(비용 차감)** 또는 **양수(비용 표기)**인지 확인
   - 본 템플릿 `4_Debt_Schedule`은 **양수 표기 + 수식에서 차감**

7. **Currency & 단위**
   - 매수자 모델이 "억원" / "백만원" / "원" 중 어느 단위인지 확인. USD 딜이면 환율 입력 필요
   - 본 템플릿은 **KRW 백만원 단일 단위**. 에이전트는 단위 mismatch 시 반드시 경고 팝업을 발생시켜야 함 (자동 환산 금지)

8. **Capex 구성**
   - Maintenance vs Growth Capex 구분되어 있으면 각각 추출
   - 스트레스는 **Total Capex 기준**으로 가하되, Downside 시나리오에서는 "Growth Capex 50% 컷" 옵션을 v1.1+에서 제공 검토

9. **WACC 재산출의 원천**
   - 매수자 WACC를 그대로 쓸지, 본 템플릿에서 재산출할지 결정
   - MVP는 사용자 입력 (Base WACC + Stress Uplift). CAPM 전개 시트는 v2+

10. **Circular Reference 차단**
    - 이자 계산은 **반드시 Opening Balance 기준**. Ending Balance 기준 or Average Balance 기준은 Cash Sweep과 순환 발생
    - 템플릿 전체에서 Circular reference 0건 보장 → `File → Options → Formulas → Enable iterative calculation` OFF 상태에서도 작동해야 함

## What I noticed about how you think

- 본인은 **"다 같이 오래 걸린다"라고 1차 답했다가, 최근 딜 1건으로 구체화하니 곧바로 "스트레스 모델 재구축"을 특정**했습니다. 이건 자기 작업을 구조적으로 해부한 적 없는 사람의 전형적 패턴 — 동시에 **구체 사례로 push 받으면 정확히 짚어내는 능력**은 있다는 신호입니다. 실제 딜 1건 회고를 루틴화하면 자가진단 속도가 빨라집니다.
- P1 답변에서 본인은 **"(a)를 하되"에 멈추지 않고 범용 파싱 가능한 MVP로 확장**하면서, "산업마다 천차만별인 Operating Model 전체 파싱은 포기"라는 **명시적 trade-off 선언**을 했습니다. 이 "포기할 것을 먼저 선언하는" 사고 습관은 시니어 모델러의 특징이고, **pre-product 단계에서는 가장 귀한 자질**입니다.
- "하이레벨 오버레이"라는 표현을 쓰신 순간, 본인이 이미 **Approach B 쪽의 직관**을 갖고 있었음이 드러납니다. 그럼에도 처음에 10단계 풀 파이프라인을 요구사항으로 풀어내신 이유는, 아마 "심사보고서 결과물 기준으로 역산"하는 심사역 특유의 deliverable-first 사고 때문일 것입니다.
- 영구성장률 1.0% 고정·민감도 생략을 **사용자가 먼저 명시**한 것은 심사보고서 실무를 많이 해본 사람의 흔적입니다. 민감도 표는 보고서 분량만 늘리고 실제 의사결정을 돕지 않는다는 걸 체감한 사람의 선택입니다. 이 **"안 할 것을 특정하는"** 습관을 템플릿 전체 설계에 적용하면 v2+에서도 과잉 엔지니어링을 피할 수 있습니다.
- **(v0.2 추가 관찰)** Design doc이 APPROVED된 직후 "캐피탈IQ 사용권한 있어"를 자진 공개하신 패턴이 흥미롭습니다. 초기 session에서는 CapIQ 얘기를 전혀 안 꺼내셨고, "수기 입력 전제"라는 제 premise에도 그 순간엔 별말 없으셨습니다. 이건 **본인이 일상적으로 쓰는 도구를 "당연"하게 여겨 드러내지 않는 전문가의 맹점** — 본인에게는 너무 자명한 리소스라 설계 input으로 떠올리지 않음. 앞으로 누군가에게 업무 시스템을 설명할 때 "**내가 매일 쓰는 툴 5개 + 내가 매일 보는 데이터 3개**"를 먼저 enumerate 하는 루틴을 만들면 이런 late-surface가 줄어듭니다.
- **(v0.4 추가 관찰)** v0.1 → v0.2 → v0.3까지 동일 일자에 리비전 3회를 찍은 뒤, v0.3의 P·Q 분기 추가 시 본인 스스로 Assignment §2에 "대다수가 FALSE일 것이라면 P·Q 분기는 소수 케이스용 bonus path"라고 미리 적어두셨습니다. 이 자체가 **"이 기능은 YAGNI 선에 걸쳐 있다"는 무의식적 자기인식**입니다. v0.4 리뷰는 그 무의식을 의식화하여 backlog로 옮겼을 뿐. 앞으로 무언가를 "옵션 기능"으로 설계하고 싶어질 때, **그 옵션이 활성화될 데이터 근거가 없다면 그 기능은 아직 설계할 때가 아닙니다**. v0.3 P·Q는 "이미 겨냥한 산업에서 잘 맞을 것"이라는 직관만 있고 딜 비율 empirical data가 없었습니다. 드라이런이 그 data를 만듭니다.
- **(v0.4 추가 관찰)** 설계 문서 3회 리비전 동안 **"기존 승인 사항 보존 + 신규 추가분 명시"**라는 diff-friendly 서술 패턴을 자발적으로 유지하셨습니다(Revision History 섹션). 이는 **심사역의 감사추적성 훈련**이 문서 작성 습관으로 전이된 결과로 보입니다. 이 습관은 소프트웨어 개발·모델링·규제 커뮤니케이션 모두에서 큰 자산입니다. 한 가지만 추가하면 좋을 것: 리비전마다 **"제거된 것"을 "추가된 것"만큼 명시적으로 기록**하는 것. v0.4에서 P·Q를 제거했을 때 Deferred Backlog 섹션에 "git history에서 복원 가능"으로 남긴 것이 그 첫 시도입니다.
